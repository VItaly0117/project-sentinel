# 2026-04-22 — Multi-instance identity patch

## Done
- Added `BOT_ID` env var to runtime config; defaults to `BYBIT_SYMBOL` for operator convenience on single-bot path.
- Scoped all `runtime_state` keys to `{bot_id}:{key}` so two bots sharing the same SQLite file no longer collide on state markers (e.g., `last_action_time`, `last_candle_timestamp`).
- Migrated all 5 event tables (signals, trades, risk_snapshots, runtime_events, error_events) to include `bot_id` column; added automatic schema migration for existing DBs.
- Updated logger name to `TradingRuntime[{bot_id}]` and bootstrap/status messages to display Bot label.
- Added 3 new tests: `test_sqlite_two_bot_ids_are_isolated_in_shared_db`, `test_load_app_config_bot_id_defaults_to_symbol`, `test_load_app_config_bot_id_can_be_set_explicitly`.
- All 73 tests pass.

## Blocked
- None.

## Decisions
- Single-bot operator path remains unchanged — no `.env` modifications needed; default behavior is transparent.
- Migration helper `_migrate_add_bot_id` uses `ALTER TABLE ADD COLUMN DEFAULT 'default'`, which is safe for existing DBs and will be ignored if column already exists.
- Instance identity is a lightweight scoping mechanism, not a full multi-bot orchestration layer — orchestration remains future work.
