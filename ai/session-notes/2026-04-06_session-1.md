# Session Note

## What changed
- Refactored live trading into `sentinel_runtime/` and kept `sentineltest.py` as a thin entrypoint.
- Removed hardcoded secrets from runtime code and moved runtime settings to `.env`-driven typed config.
- Added MVP runtime safeguards: closed-candle-only decisions, retry/backoff, API circuit breaker, daily loss stop, drawdown stop, reserve balance check, and open position/order limits.
- Refactored training into `sentinel_training/` and kept `train_v4.py` as a thin entrypoint.
- Replaced the old train/test flow with explicit time-aware train/validation/test splits plus optional purge gap and embargo.
- Added run-scoped training artifacts with model and metadata output under `artifacts/train_v4/`.
- Updated project memory files in `ai/`.

## What decisions were made
- Preserve strategy intent by reusing the same feature engineering logic through the shared `SMCEngine`.
- Keep backward compatibility where practical by preserving `sentineltest.py`, `train_v4.py`, and optional `monster_v4_2.json` output.
- Keep both refactors MVP-sized: modular code, explicit config, safer defaults, no web panel, no platform-wide rewrite.
- Default runtime mode is non-live; live mode requires explicit opt-in.
- Validation is used for early stopping; the final test split is held out from fitting and tuning.

## Open risks
- No automated tests exist yet for runtime safeguards or training split boundaries.
- Runtime risk baselines are not persisted across restarts unless `STARTING_BALANCE` is provided.
- Research labels still rely on OHLC execution assumptions and ignore slippage, spread, latency, and order book effects.
- The project still has no DB, Redis, CI/CD, Docker deployment, admin panel, or analyst pipeline.
