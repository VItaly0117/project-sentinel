# Session Note

## What changed
- Added `sentinel_runtime/storage.py` with a lightweight SQLite persistence layer.
- Integrated runtime persistence for signals, opened/closed trades, risk snapshots, runtime events, error events, and runtime state markers.
- Persisted `last_processed_candle_time`, `last_reported_closed_trade_id`, and baseline balance so restarts are safer and easier to inspect.
- Added local DB configuration and README notes for inspecting the SQLite file.
- Expanded runtime pytest coverage to include persistence/state behavior.

## What decisions were made
- Used stdlib `sqlite3` with explicit SQL tables instead of introducing an ORM.
- Kept persistence local and MVP-sized: one DB file, clear interfaces, no platform-wide data model.
- Preserved the current module split and kept `sentineltest.py` as a thin entrypoint.
- Stored decimals as text and timestamps as ISO strings for simple, inspectable SQLite rows.

## Open risks
- Persistence is single-node local SQLite only; there is no replication, migration flow, or remote durability.
- Runtime still depends on a local `monster_v4_2.json`.
- Training pipeline still has no automated tests.
- The project still lacks DB/Redis-backed platform services, CI/CD, Docker deployment, admin panel, and analyst pipeline.

## Next recommended task
- Add a focused pytest suite for `sentinel_training/`, especially timestamp parsing, purge/embargo split integrity, and artifact metadata output.
